# v2.0 Scope — Browser/WASM Frontend + Performance

Status: **Scoped — de-risking spikes S1–S4 complete** (research reports
[07](research/07-canvas-text-rendering.md) · [08](research/08-wasm-toolchain-size.md) ·
[09](research/09-simd-parallel-raster.md) · [10](research/10-wasm-threads-hosting.md)).
Target version: **2.0.0**. Builds on the v1.0 MVP; language-level grounding in
[docs/research/04-language-evaluation.md](research/04-language-evaluation.md).

## 1. Theme

v2.0 delivers on the two original headline goals the MVP deferred:

1. **"Portable core, two frontends"** — compile `tte-core` to WebAssembly and add a
   **browser frontend** that renders the same character grid to an HTML canvas. The
   engine becomes shareable by URL, not just runnable in a terminal.
2. **"Performant"** — a **performance push**: tile-based multithreaded rasterization
   and a SIMD inner loop, with speedups tracked in CI.

> **Pairing assumption:** the v2.0 milestone bundles **WASM frontend (Phase 5)** +
> **Performance (Phase 6)**. They reinforce each other — WASM SIMD/threads are exactly
> the perf primitives report 04 evaluated. If the intended second theme was instead
> *rendering fidelity* or *DSL/UX*, swap Phase 6 and this scope still stands.

## 2. Why these two, now

- The whole architecture was built **library-first** precisely so a second frontend
  could slot in without touching the engine (project brief D1/D4: rendering produces a
  pure `Framebuffer`/`CellBuffer`; presentation is a swappable layer). Phase 5 cashes
  that in and *proves the thesis*.
- Performance was the stated headline from day one. The MVP correctly deferred it
  (research 05: at terminal cell counts, I/O bound, not raster bound) — but the **browser
  canvas and high-resolution half-block modes raise effective pixel counts** to where
  SIMD + threads matter. Phase 6 lands when there's finally a reason for it.

## 3. Key decisions

| # | Decision | Rationale |
|---|---|---|
| V1 | **Keep the "text out" identity in the browser**: the canvas draws the cell grid as monospace glyphs with per-cell color (ASCILINE-style), not a switch to pixel rendering. Half-block mode already gives pixel-like fidelity while staying "text". | Stays true to the project's identity; reuses the existing presenter seam. |
| V2 | **New `tte-wasm` cdylib crate** with a thin `wasm-bindgen` API over `tte-core`; all logic stays in the (already-tested) core. | Mirrors the `tte-cli` thin-frontend pattern; keeps wasm glue minimal. |
| V3 | **Browser frame export lives in core** as a pure function (`Framebuffer` → compact per-cell `{glyph, fg, bg}` buffer), so it's unit-tested natively and the wasm layer only marshals bytes. | Same testability seam as the terminal presenters (docs/02 §4). |
| V4 | **Toolchain: pinned `cargo build --target wasm32-unknown-unknown` → `wasm-bindgen` CLI → `wasm-opt -Oz`** (wasm-pack acceptable as the convenience wrapper); static `web/` demo (index.html + JS glue), no framework. Binary-size budget tracked with `twiggy`. | Report 08: rustwasm org sunset (Sept 2025) but wasm-bindgen/wasm-pack live on under the `wasm-bindgen` org; the explicit pipeline avoids wrapper rot and pins CLI == crate version. |
| V4a | **`web-sys` stays OUT of `tte-wasm`**: the wasm module exposes only data (typed-array frames); all DOM/canvas work lives in hand-written JS. | Report 08: smaller binary, simpler glue, Node-only wasm tests, reusable core — the consensus hybrid pattern. |
| V5 | **Phase 5 ships single-threaded, stable wasm** (wasm SIMD128 optional, no threads). **Threads are a Phase 6 stretch** via a separate nightly build (wasm-bindgen-rayon + COOP/COEP headers). rayon's serial fallback on wasm means no code fork for this. | Reports 04/09/10: wasm threads still need a pinned nightly + build-std + cross-origin isolation, and upstream wasm-bindgen-rayon is a personal-fork dependency — real but opt-in cost; don't block the frontend on it. |
| V6 | **Performance is feature-gated and behavior-preserving**: `parallel` (rayon tiles, order-stable binning) and `simd` (the `wide` crate — one stable code path across x86-64/aarch64/wasm-simd128) features; **coverage moves to integer edge functions** (Giesen `orient2d`), making SIMD↔scalar byte-parity trivial *and* edges watertight; **output must stay byte-identical to the scalar path** (golden frames are the regression guard). | Report 09: integer edge functions buy determinism + seam correctness in one change; `wide`'s fixed lane count gives host-independent op order; `std::simd` still nightly. |
| V7 | **Native build enables `parallel` by default; wasm uses scalar** (stable wasm has no threads) unless the threaded nightly variant is built. | Keeps both targets green on stable. |
| V8 | **Browser renderer: Canvas2D + glyph atlas, full redraw every rAF** — pre-render the glyph set to an offscreen canvas, `drawImage` per cell at integer coords; half-block bg via batched `fillRect`/`putImageData`. No dirty-cell diffing (worthless under full-frame 3D animation); WebGL deferred behind a renderer interface (trigger: the 200×100 stretch); WebGPU skipped. | Report 07: measured ~0.6 µs/cell (xterm.js PR #1790) ⇒ ~4–5 ms/frame at 120×60 — 6× under the 30 FPS budget. |

## 4. Phase 5 — Browser / WASM frontend (the headline)

**Deliverable:** open a web page, see a model/scene rendered as a colored character grid
on a canvas, orbit it with the mouse, and edit a scene live in a textarea.

| ID | Requirement |
|---|---|
| FR-5.1 | `tte-wasm` cdylib exposing a `wasm-bindgen` `Renderer`: construct with cell `width`/`height`; `load_obj(text)` and `load_scene(text)` (DSL); `set_orbit(yaw, pitch, radius)`; `render()`. Parse/DSL errors surface as JS exceptions with the line-numbered message. |
| FR-5.2 | Core browser-frame export: a pure `tte-core` function `Framebuffer` → a compact per-cell representation (glyph + packed `fg`/`bg` RGB), unit-tested natively. The wasm layer returns it as typed arrays (no per-cell JS calls). |
| FR-5.3 | Browser frontend (`web/`): a canvas drawing the cell grid via the **glyph-atlas renderer (V8)**; **mouse-drag orbit + wheel zoom AND touch support (one-finger drag orbit + pinch zoom)** mapped to `OrbitCamera` via Pointer Events; a **live scene editor** (textarea → re-render on change, browser-side hot-reload); a few built-in presets (cube, sphere, sample scene). |
| FR-5.4 | Build pipeline per **V4**: documented build script producing the wasm module + JS bindings + a static, servable `web/` directory (works offline, no CDN deps; `.nojekyll` for GitHub Pages, `application/wasm` MIME noted for local serving). |
| FR-5.5 | `wasm-bindgen-test` smoke test: construct a `Renderer`, load the cube, `render()`, assert a non-empty, expected-dimension frame; runs under **Node** (`wasm-pack test --node` — no DOM in the wasm crate per V4a) and is wired into a CI job. |
| FR-5.6 | Frame buffers are **pre-allocated at `Renderer` construction** so steady-state `render()` never grows wasm memory (keeps typed-array views/copies stable and the boundary cheap at 60 fps; report 08 Q4). |

## 5. Phase 6 — Performance push

**Deliverable:** the same frames, produced measurably faster on multi-core CPUs and via
SIMD, with the speedups visible in `cargo bench` / CI.

| ID | Requirement |
|---|---|
| FR-6.1 | **Tile-based multithreaded rasterization** (native): partition the framebuffer into tiles, rasterize triangles into tiles in parallel with `rayon`; race-free by tile ownership. Behind a `parallel` feature (on by default natively, off for stable wasm). |
| FR-6.0 | **Integer edge-function coverage** (prerequisite, V6): move triangle coverage from float barycentrics to Giesen-style `i32 orient2d` with a top-left fill rule, in the *scalar* path first. Exact integer math ⇒ SIMD parity becomes trivial and shared edges become watertight (replaces the inclusive-coverage seam workaround). Golden frames re-approved once, deliberately, for this change. |
| FR-6.2 | **SIMD inner loop**: vectorize edge-function coverage + attribute interpolation with the **`wide` crate** (4-px spans, fixed lane count, no FMA contraction in interpolation), behind a `simd` feature with a scalar fallback — one code path for x86-64/aarch64/wasm-simd128. |
| FR-6.3 | **Behavior preservation**: parallel and SIMD paths produce **byte-identical** output to the scalar path — asserted by running the existing golden-frame suite against every feature combination in CI. Parallel determinism via per-tile triangle lists binned in ascending global index (per-tile draw order == scalar order ⇒ identical strict-`<` z tie-breaks). |
| FR-6.4 | **Benchmark expansion**: criterion benches over large meshes (≥100k triangles) and high cell resolutions (e.g. 400×200 half-block); scalar vs `parallel` vs `simd` comparisons; baselines tracked as CI artifacts. |
| FR-6.5 *(stretch)* | **WASM threads**: a nightly build variant using `wasm-bindgen-rayon` + a demo served with COOP/COEP headers, parallelizing the browser rasterizer. Clearly separated from the stable default build. |

### Phase 6 non-functional

| ID | Requirement | Verification |
|---|---|---|
| NFR-10 | `parallel` gives a **≥3× speedup** over scalar on an 8-core CPU, **measured on the large bench** (≥100k tris @ 400×200, FR-6.4) — the small 1k-tri/200×50 working set won't show it (report 09 Q7). | criterion comparison |
| NFR-11 | `simd` gives a **≥2× speedup** on the rasterizer fill stage (single-thread). Realistic per report 09 (2–3× typical); short spans on tiny triangles erode it, hence fill-stage micro-bench. | criterion micro-bench |
| NFR-12 | All feature combinations (`--no-default-features`, `parallel`, `simd`, both) pass the full test + golden-frame suite. | CI feature matrix |

## 6. Verification strategy

- **Golden frames are the safety net for the whole release**: every perf path and the
  WASM export must reproduce the v1.0 golden frames byte-for-byte. This is the strongest
  possible regression guard and is already in place (docs/02 §3).
- **WASM**: `wasm-bindgen-test` for the glue; the heavy logic is covered by existing
  native `tte-core` tests. A shared golden frame compares native vs wasm output.
- **CI grows**: a `wasm` job (`wasm-pack test` + size check) and a `features` matrix job
  (scalar/parallel/simd) join the existing fmt/clippy/test/deny pipeline.

## 8. Risks

1. **WASM threads' cross-origin isolation** (COOP/COEP) complicates hosting and is
   nightly-only with a personal-fork dependency (report 10). *Mitigation:* keep it a
   clearly-separated stretch (FR-6.5); the default browser build is stable + scalar and
   ships regardless. GitHub Pages cannot set the headers — the stretch demo, if built,
   moves to Netlify/Cloudflare Pages or uses `coi-serviceworker`.
2. **SIMD/parallel determinism drift** — ~~resolved by design~~: integer edge functions
   (FR-6.0) make coverage exact, and order-stable tile binning preserves z tie-breaks
   (report 09). Residual risk is only in float *interpolation* ordering; FR-6.3 keeps
   byte-identical output a hard requirement — treat any diff as a bug, not a snapshot
   update.
3. **Canvas text-rendering throughput** — ~~retired~~: measured numbers (report 07) put
   the Canvas2D glyph atlas 6× under the NFR-8 budget at 120×60. Only the 200×100
   stretch is tight; the renderer interface (V8) keeps a WebGL path open.
4. **Binary-size creep** — substantially de-risked (report 08): no web-sys (V4a), no
   serde, panic=abort + wasm-opt -Oz; comparable projects land tens of KB. NFR-7 CI
   check remains as the guard. Watch `format!`-heavy error paths in the DSL parser.
5. **NEW — small-working-set parallelism** (report 09): at 1k tris / 200×50 the
   parallel path may show little-to-no gain (overheads dominate). *Mitigation:* NFR-10
   is measured on the large bench by definition; consider a triangle-count threshold
   below which the scalar path runs even with `parallel` enabled.

## 9. Spike outcomes (research complete)

All four de-risking spikes ran before implementation (reports 07–10):

| Spike | Question | Outcome |
|---|---|---|
| S1 | Canvas renderer for ≥30 FPS at 120×60? | **PASS, 6× margin** — Canvas2D + glyph atlas, ~0.6 µs/cell measured (xterm.js); full redraw per rAF, no dirty-diffing; WebGL deferred, WebGPU skipped → V8, FR-5.3 |
| S2 | Toolchain health + ≤250 KB binary? | **PASS** — rustwasm org sunset but wasm-bindgen/wasm-pack maintained under new org; pinned cargo→wasm-bindgen→wasm-opt pipeline; web-sys excluded; tens-of-KB realistic → V4/V4a, FR-5.4–5.6, NFR-7 |
| S3 | Stable SIMD + deterministic parallelism? | **PASS with a design change** — `wide` crate (one path: x86-64/aarch64/wasm-simd128); **integer edge functions** make SIMD parity trivial + edges watertight (new FR-6.0); order-stable tile binning ⇒ bit-identical parallel output; NFR-10/11 realistic with the small-working-set caveat |
| S4 | WASM threads worth it? | **KEEP AS STRETCH** — still nightly + build-std; wasm-bindgen-rayon upstream archived (live personal fork); GitHub Pages can't set COOP/COEP; payoff bounded (1.5–3×). Single-thread fallback already required, so the stretch stays purely additive → FR-6.5 unchanged |
