# Research 09: SIMD + Parallel Rasterization (spikes S3, partial S4)

Date: 2026-06-12. Validates v2.0 Phase 6 (FR-6.1..6.5, NFR-10..12). Status: IN PROGRESS — findings appended incrementally.

## Q1. Stable-Rust portable SIMD options (2025/2026)

- **`wide` (Lokathor)**: mature, supports x86 SSE/AVX, NEON, **and wasm simd128**; fixed-width types (`f32x4`, `f32x8`, `i32x4`/`i32x8`) that lower to native instructions or scalar fallback. **No multiversioning** (compile-time feature detection only — you pick target features at build time, e.g. `RUSTFLAGS=-C target-feature=+simd128` for wasm). Source: Shnatsel, "The state of SIMD in Rust in 2025" (https://shnatsel.medium.com/the-state-of-simd-in-rust-in-2025-32c263e5f53d, summarized via search snippets; article itself blocks fetch). Confidence: HIGH (corroborated by crates.io/github listing https://github.com/Lokathor/wide).
- **`pulp` (sarah-quinones / used by faer)**: higher-level, **built-in runtime multiversioning** (per-CPU dispatch), but operates on *native SIMD width* — your kernel must be generic over lane count; arch support per the 2025 state-of-SIMD article is **NEON, AVX2, AVX-512 only** (wasm simd128 support unclear/limited). Confidence: MEDIUM (snippet-level; verify against pulp docs below).
- **`std::simd`**: still **nightly-only** (`#![feature(portable_simd)]`) as of the 2025 survey; not usable under our stable-Rust constraint. Confidence: MEDIUM-HIGH (to re-verify).
- **`std::arch` intrinsics + `is_x86_feature_detected!`**: stable, fastest ceiling, but requires per-arch code (x86 SSE2/AVX2, aarch64 NEON, wasm `core::arch::wasm32` simd128 intrinsics) = 3 codepaths + scalar. High maintenance for a small kernel.
- Preliminary verdict for our kernel: **`wide`** — fixed-width `i32x4`/`f32x4` matches a 4-pixel-span edge-function kernel exactly, compiles on all three targets (x86-64, aarch64, wasm32+simd128) from one code path, stable Rust, zero unsafe. Multiversioning loss is acceptable: SSE2 is baseline on x86-64, NEON baseline on aarch64, simd128 is a build flag on wasm. `pulp`'s variable-width model is awkward for a tiny 4/8-pixel raster kernel and wasm support is doubtful.

## Context anchors (local)

- FR-6.2 already names `wide`/`std::arch` (stable, not nightly std::simd); FR-6.3 = byte-identical output across feature combos; NFR-10 ≥3× parallel (8-core), NFR-11 ≥2× SIMD fill-stage; V5/V7: wasm ships scalar single-thread on stable, wasm-bindgen-rayon is a nightly stretch (FR-6.5).

## Q1 (addendum). New contenders + verification (2026)

- **`std::simd` still NIGHTLY in 2026.** No stabilization landed; `doc.rust-lang.org/std/simd` still gated on `#![feature(portable_simd)]` (tracking #86656). Confirms FR-6.2's choice to avoid it. Confidence: HIGH. (https://doc.rust-lang.org/std/simd/index.html, https://pythonspeed.com/articles/simd-stable-rust/)
- **`fearless_simd` v0.2.0 (Linebender, Raph Levien)** — first release since 2018, portable SIMD across **WASM, aarch64, x86/x86_64** on stable; goal is a safe portable layer like a stable-Rust std::simd. New/young (0.2.x), API churn expected. Worth watching but too immature to bet a byte-identical golden-frame contract on in 2026. Confidence: MEDIUM. (https://linebender.org/blog/tmil-19/, state-of-SIMD 2025)
- **`pulp`** — Linebender's sparse-strip renderers (vello_cpu) drove **strong WASM SIMD + NEON** support into pulp through 2025; multiversioning is real and proven (powers faer). BUT pulp's model is *native-width-generic* (you write lane-count-agnostic kernels), which is awkward for a tiny fixed 4/8-pixel edge-function kernel, and the "byte-identical across feature combos" contract is harder when the dispatched width varies at runtime (AVX2 8-wide vs SSE 4-wide vs simd128 4-wide could reorder reductions). Confidence: MEDIUM-HIGH.
- **Verdict unchanged: `wide`.** Fixed-width `i32x4`/`f32x4` = one code path on all 3 targets, stable, no unsafe, and — critically — **fixed lane count means identical op order regardless of host CPU**, which is what the golden-frame contract needs. Multiversioning (pulp's edge) is irrelevant here: SSE2 baseline on x86-64, NEON baseline on aarch64, simd128 a build flag on wasm. Use `f32x4` only if needed; prefer **`i32x4` integer edge functions** (see Q3).

## Q2. Canonical SIMD rasterization (Giesen) — verified

Giesen "Optimizing the basic rasterizer" / "Triangle rasterization in practice" (fetched via rygorous/rygblog-src mirror; wordpress + medium 403 on fetch):
- **Edge function is integer affine**: `orient2d(a,b,c) = (b.x-a.x)*(c.y-a.y) - (b.y-a.y)*(c.x-a.x)`, i.e. `F(x,y)=A·x+B·y+C` with `A=(a.y-b.y)`, `B=(b.x-a.x)`, integer coefficients from (sub-pixel snapped) integer vertex coords.
- **Incremental stepping**: stepping +1 px in x adds A; +1 px in y adds B. Per-pixel cost collapses from "2 mul + 5 sub per edge" to **3 integer adds** (one per edge). Full eval only once per triangle (or per tile corner).
- **SIMD over a span**: broadcast A/B/C, init 4 (or 8) consecutive x lanes, step delta becomes `i32x4(A * 4)`. Loop body is unchanged vs scalar — "minimal code changes." Process 4/8 pixels per iteration.
- **Masked depth write**: inside-test = sign bits of the 3 edge SIMD values OR'd; that mask AND'd with `z < zbuf` mask gives the write mask; blend/masked-store updates depth + color per lane (no scalar branch). 
- **Speedups**: Giesen does not give a clean isolated multiplier in these posts (the full occlusion-culling series reports large cumulative gains from tiling+SIMD+threading together). krzosa rasterizer: inner loop fully vectorized AVX2/FMA, **8 px/iter**, color channels kept in separate registers (8 reds, etc.). Expect ~3–6× on the fill stage from 8-wide (less than 8× due to edge/partial tiles + memory). Confidence: HIGH on technique, MEDIUM on exact multiplier.

## Q3. DETERMINISM — the decisive finding

- **Float SIMD CAN diverge from scalar**: different association order in reductions, and **FMA contraction** (`a*b+c` fused → different rounding) produce non-bit-identical results. Rust does NOT auto-contract FMA in release by default for separate `*`/`+` (no `-ffast-math` equiv on stable), but `mul_add` and some autovectorization patterns can. So a *float* edge/barycentric SIMD path is a byte-identical liability.
- **Integer edge functions make parity TRIVIAL** (and are Giesen's recommended approach). `orient2d` and incremental stepping are **pure i32/i64 arithmetic — exact, associative, platform-independent, no rounding**. A 4-lane `i32x4` edge evaluator produces *bit-for-bit* the same coverage mask as the scalar i32 path on x86, aarch64, and wasm. This makes FR-6.3 (byte-identical) essentially free for the *coverage* stage.
- **Watertight as a bonus**: integer edge funcs + **top-left fill rule** (bias 0 for top/left edges, −1 otherwise) guarantee "every pixel covered by two non-overlapping triangles sharing an edge is lit once and only once" — no cracks/seams, no double-cover. This simultaneously fixes seams AND gives determinism. STRONG recommendation: move coverage to integer edge functions for v2.0.
- **Intensity/z interpolation**: barycentric weights derived from the same integer edge values (`w_i = F_i / area`). If intensity/z stay **float**, keep the *exact same scalar op sequence* in the SIMD lanes (same divides, same multiply order, NO `mul_add`/FMA) so each lane reproduces the scalar bit pattern. Alternative: fixed-point interpolation = trivially identical but needs care on z precision. Verdict: integer coverage + carefully-ordered non-FMA float interpolation = byte-identical achievable; FR-6.3 is FEASIBLE. Confidence: HIGH.

Sources: https://github.com/rygorous/rygblog-src/blob/master/posts/optimizing-the-basic-rasterizer.md , https://raw.githubusercontent.com/rygorous/rygblog-src/master/posts/triangle-rasterization-in-practice.md , https://github.com/krzosa/software_rasterizer

## Q6. rayon on wasm + wasm-bindgen-rayon (verified)

- **rayon builds + runs CLEANLY on wasm32 with a single-threaded fallback.** When the global thread pool can't spawn threads (thread creation returns `io::ErrorKind::Unsupported`, as on `wasm32-unknown-unknown` / `wasm32-wasi`), rayon does NOT panic — it configures a fallback that behaves like `RAYON_NUM_THREADS=1`: `join` runs its two closures sequentially, `par_iter()` executes in-order on the calling thread. So our `parallel` feature's `par_iter`-over-tiles code **compiles and runs unchanged on wasm**, just serially. CAVEAT: the fallback only applies to the *global/implicit* pool; explicit `ThreadPoolBuilder` always surfaces the Unsupported error (don't build a custom pool on wasm). This means V5/V7 "wasm ships scalar single-thread" is automatic — the same parallel code path degrades gracefully, no `cfg(target_arch=wasm32)` fork needed for correctness. Confidence: HIGH. (https://docs.rs/rayon-core / rayon discussion #1155, issue #861)
- **wasm-bindgen-rayon (threaded stretch, FR-6.5) is NIGHTLY-ONLY and getting more fragile, not less.** It requires a nightly toolchain to rebuild `std` with `-Z build-std` + atomics/bulk-memory target features, plus a JS-side worker-pool init and cross-origin isolation (COOP/COEP headers). 2026 status: still maintained but the canonical repo moved (GoogleChromeLabs → RReverser/`wasm-bindgen-rayon`), tested against pinned nightlies (e.g. nightly-2025-11-15), and as of nightly 2026-05-06 it additionally needs `-Clink-arg=--export=__heap_base` in RUSTFLAGS. WASM threads are still NOT first-class in Rust (nightly only). Broader context: the rustwasm GitHub org was sunset/archived in Sept 2025; `wasm-bindgen` itself survived under a new org with new maintainers, but the surrounding tooling is in flux. Verdict: keep wasm-threads strictly a **pinned-nightly experimental stretch**, never on the stable golden-frame CI path. Confidence: HIGH. (https://github.com/RReverser/wasm-bindgen-rayon, https://blog.rust-lang.org/inside-rust/2025/07/21/sunsetting-the-rustwasm-github-org/, https://docs.rs/wasm-bindgen-rayon)

## Q4. Tile-based rayon design (verified)

- **Architecture**: split the framebuffer into tiles (e.g. 16×16 or 32×8 px; Intel MOC uses 32×8 to fill a 256-bit AVX2 coverage mask). Phase 1 = **bin**: for each triangle, compute its bbox→tile range, push its index into each overlapped tile's list. Phase 2 = **`par_iter()` over tiles**: each tile owns a disjoint slice of color+depth buffers, so threads NEVER touch the same pixel → **no atomics, no locks, no data races** on the framebuffer. This is the standard CPU-rasterizer parallelization (Intel MOC, GameTechDev/OcclusionCulling, krzosa, zachbethel, vblanco20-1). Confidence: HIGH.
- **Why par-over-triangles + atomic-z is BAD**: two triangles can cover the same pixel, so concurrent threads race on that pixel's depth+color. An atomic z-test (CAS loop: read z, compare, conditionally write z AND color) is (a) a correctness hazard — z and color aren't one atomic unit, so you get torn writes / wrong color winning unless you pack them or lock; (b) slow — contended atomics serialize; (c) **non-deterministic** — the winner at equal/racing depth depends on thread timing. Tiling sidesteps all three by making each pixel single-owner. Confidence: HIGH.
- **Binning cost**: extra pass over triangles computing bbox + tile-list pushes; cheap relative to fill, but allocates per-tile lists (use a flat bump/SmallVec or per-tile Vec). For large tris spanning many tiles the index is duplicated across lists (memory, not much compute). At our scale (≤100k tris) binning is negligible vs fill.
- **Speedup numbers**: zachbethel multithreaded tile rasterizer = **~4× with 8 threads on a 4-core+HT i7** (good 1→8 scaling, ~physical-core-bound). Intel software-occlusion sample = **up to 8× overall** (SSE/AVX + multithreading + frustum/occlusion culling combined). rawrunprotected = **2–3× single-thread from AVX alone**. krzosa/vblanco = fully AVX2/FMA-vectorized, ~870k tris @1024² in ~20ms on Ryzen 1700X (8-core). Takeaway: parallel speedup tracks **physical cores** (HT adds little for this ALU+memory-bound work); expect ~0.5–0.8× efficiency, i.e. **~4–6× on 8 physical cores** realistically. Confidence: HIGH on pattern, MEDIUM on exact ratio for our tiny resolution.

## Q5. Determinism under parallelism (verified argument)

- **Tiling gives byte-identical output to scalar IF two invariants hold**: (1) **single-owner pixels** — each pixel written by exactly one tile/thread (guaranteed by disjoint tile slices), so there are zero inter-thread races to order; (2) **fixed per-tile triangle order** — within a tile, draw triangles in a deterministic order (e.g. ascending global triangle index), the SAME order the scalar path uses. Because rayon parallelizes ACROSS tiles (independent) but each tile is processed SEQUENTIALLY internally, the per-pixel sequence of (z-test, conditional write) is identical to scalar. Result: bit-identical framebuffer regardless of thread count or scheduling. Confidence: HIGH.
- **Equal-depth tie-break is the subtle point**: with `z < zbuf` (strict), the first-drawn triangle at a given depth wins and later equal-z ones are rejected — so the WINNER depends on draw order. As long as per-tile order == scalar global order, the tie-break resolves identically. Binning MUST preserve order: push triangle indices into tile lists in ascending index, and iterate each tile's list in that order. (If binning is itself parallel, it must produce order-stable lists — e.g. sort each tile list by index, or bin serially.) This is the one place a careless implementation breaks the golden-frame contract. Confidence: HIGH.
- Combined with Q3's **integer edge functions** (coverage is exact i32, platform-independent) and non-FMA ordered float interpolation, the `scalar | simd | parallel | simd+parallel` matrix is all byte-identical by construction. FR-6.3 holds.

## Q7. Our scale — where does time go, are NFR-10/11 realistic?

- Baseline: ~111µs for 1k tris @200×50 (10k cells), scalar single-thread. That's tiny tris (avg ~10 cells/tri) over a small buffer.
- **Time breakdown (inferred)**: at this scale work splits roughly into (a) per-triangle **transform/setup** (vertex transform, edge-coeff setup, bbox) — O(tris), and (b) per-pixel **fill** (edge eval + z-test + intensity interp) — O(covered cells). With ~10 cells/tri, setup is a LARGE fraction (setup is per-tri fixed cost amortized over only ~10 px). At 100k tris the buffer (≤400×200=80k cells) is overdrawn many times → fill dominates and setup is amortized better. So the transform/setup vs fill ratio is scene-dependent; **small-tri/low-overdraw scenes are setup-bound, large/high-overdraw scenes are fill-bound**.
- **NFR-11 (≥2× SIMD on fill stage)**: REALISTIC but only for the **fill** stage and only when fill dominates. 4-wide `i32x4` over a span gives ~2–4× on fill (Giesen "minimal code change," rawrunprotected 2–3× from AVX, Intel 3×). For tiny ~10-cell tris, spans are short (many partial 4-px groups), eroding the win — the 2× target is the floor and is plausible on the fill stage specifically, NOT on end-to-end frame time if setup-bound. SIMD-ing the per-triangle setup (4 triangles' coeffs at once) helps the small-tri case. Confidence: MEDIUM-HIGH.
- **NFR-10 (≥3× parallel on 8 cores)**: REALISTIC. zachbethel got 4× on 4-core+HT; 8 physical cores at even 0.5 efficiency = 4×. The risk is the buffer is so SMALL (10k–80k cells) that per-tile work is tiny and rayon's task overhead + binning eats into it at 1k tris. Mitigation: coarse tiles, batch tiles per task, and note the 3× target is easiest at the **100k-tri / 400×200 end** where there's enough work to amortize scheduling. At the **1k-tri / 200×50 end** parallel speedup may be <3× due to overhead — flag this. Confidence: MEDIUM (small working set is the main threat to NFR-10).

## Recommendation (closes S3 + S4; verdict on FR-6.x / NFR-10/11)

- **SIMD crate + kernel (S3):** Use **`wide`** (`i32x4`/`f32x4`), one code path on x86-64 + aarch64 + wasm32-simd128, stable, no `unsafe`, fixed lane count ⇒ host-independent op order. Reject `std::simd` (still nightly 2026), `pulp`/`fearless_simd` (native-width-generic or too young — both complicate the byte-identical contract). Kernel: process a **4-pixel span** per iteration — broadcast edge coeffs A/B/C, init 4 x-lanes, step by `i32x4(4*A)`; inside-mask = OR of the 3 edge sign bits; AND with `z<zbuf` mask; masked-blend depth+intensity. SIMD the per-triangle **setup** too (4 tris' coeffs at once) to help the small-tri/setup-bound case.
- **Coverage = INTEGER edge functions (the decisive call).** Move coverage from float to Giesen's i32 `orient2d` + incremental stepping. This makes SIMD↔scalar parity **trivial and exact** (integer math is associative, no rounding, no FMA, platform-independent) AND, with a **top-left fill rule**, makes triangle edges watertight (kills seams). One change buys both determinism and correctness. **Strong yes on integer edge functions.**
- **Determinism strategy (FR-6.3):** integer coverage (exact) + intensity/z interpolation that **replays the exact scalar float op order with NO `mul_add`/FMA contraction** per lane (or go fixed-point for trivial parity). Across the `scalar|simd|parallel|simd+parallel` matrix, output is byte-identical by construction (NFR-12 / golden frames hold).
- **Tile design (FR-6.1):** framebuffer → tiles (start 16×16; tune); Phase 1 bin triangle indices into per-tile lists in **ascending global index**; Phase 2 `par_iter()` over tiles, each tile owns a disjoint fb slice ⇒ **no atomics, no races**. Per tile, draw in ascending index (= scalar order) so the strict-`z<` equal-depth tie-break resolves identically ⇒ **bit-identical to scalar**. NEVER do par-over-triangles + atomic-z (racy, slow, non-deterministic). If binning is parallelized, keep tile lists order-stable.
- **wasm-threads verdict (S4 / FR-6.5):** keep it a **pinned-nightly experimental stretch only**. Two facts: (1) **rayon already degrades cleanly to single-thread on wasm** (Unsupported→global fallback, `par_iter` runs serially) — so the same `parallel` code compiles+runs on stable wasm with zero forking, satisfying V5/V7 automatically; (2) real wasm threads need `wasm-bindgen-rayon` = nightly `-Z build-std` + atomics + COOP/COEP + (since nightly 2026-05-06) `--export=__heap_base`, on a repo that just changed orgs. Ship stable scalar wasm; gate threads behind a separate pinned-nightly build, off the golden-frame CI path.
- **Are NFR-10/11 realistic?** **NFR-11 (≥2× SIMD fill): YES** for the fill stage with 4-wide integer edge eval (lit corroboration: Giesen "minimal change," rawrunprotected 2–3× AVX single-thread, Intel 3×); 2× is a safe floor though short spans on ~10-cell tris erode it — benchmark on the 400×200/large-mesh end. **NFR-10 (≥3× parallel, 8-core): YES at the high end** (zachbethel 4× on 4-core+HT ⇒ ≥4× on 8 physical cores expected). **RISK: the low end** (1k tris @200×50 = 10k cells) has too little work — rayon task + binning overhead may keep it <3×. Mitigation: define NFR-10 against the **≥100k-tri / 400×200** bench (FR-6.4 already specifies this), use coarse tiles / batch tiles per task. Both targets are achievable as written; flag the small-working-set caveat for NFR-10.
